# Simplified PRD Template

---

## Instructions for AI Model

**Before generating the PRD, the AI model MUST:**

### 1. Ask Clarifying Questions

Ask only the **essential** questions to get started. The AI should handle the heavy lifting and make intelligent defaults:

**Core Questions (Ask these first):**
1. **What is the project name and what does it do?** (1-2 sentences)
2. **Who will use this product?** (primary user type)
3. **What are the 3 must-have features for MVP?**
4. **How much time do you have?** (days/weeks)

**Optional Follow-ups (Only if needed):**
- Is there a specific mood or feeling you want? (e.g., professional, playful, modern)
- Any color preferences? (The AI will suggest a palette if none provided)
- Any technical requirements or constraints? (The AI will use modern best practices by default)

**The AI Should Automatically Handle:**
- ✅ Selecting appropriate technology stack based on project type
- ✅ Creating detailed user personas from brief descriptions
- ✅ Expanding 3 features into complete MVP scope
- ✅ Generating complete color palette from mood/preferences
- ✅ Determining team size and skill level from timeline
- ✅ Suggesting design system and components
- ✅ Creating accessibility requirements
- ✅ Defining responsive breakpoints

---

### 2. Confirm Understanding

**Briefly summarize in 3-4 bullet points:**
- Project: [Name] - [What it does]
- Users: [Who uses it]
- MVP: [3 core features]
- Timeline: [Duration] 
- Style: [Design direction - AI will suggest if not specified]

**Then ask:** "Does this sound right? Should I proceed with the full PRD?"

---

### 3. Generate Complete PRD

**Only after confirmation**, create the comprehensive PRD with:
- Full branding, theming & color system (OKLCH)
- Complete component inventory
- Detailed wireframe structure
- All sections filled with intelligent defaults

**The AI must make the process smooth by:**
- Making reasonable assumptions based on project type
- Providing complete, production-ready specifications
- Offering 2-3 alternatives for subjective choices (colors, fonts)
- Using industry best practices as defaults
- Being opinionated where it helps (but explaining why)

---

**Note:** The goal is to reduce cognitive load on the student. The AI should be **proactive**, not just reactive. If information is missing, make educated guesses and note them in the PRD with: `[AI Suggestion: ... You can change this to ...]`

### Color System (OKLCH)

**Understanding OKLCH:**
OKLCH is a perceptual color space that provides:
- **L (Lightness):** 0-100% (0 = black, 100 = white)
- **C (Chroma):** 0-0.4 (0 = grayscale, higher = more vibrant)
- **H (Hue):** 0-360 degrees (color wheel position)

**Color Palette Definition:**

#### Primary Brand Color
```css
--color-primary: oklch([L]% [C] [H]);
--color-primary-content: oklch([L]% [C] [H]);
```

**Example:** `oklch(60% 0.22 250)` - Vibrant blue
- **Usage:** Primary CTAs, links, brand moments, key actions
- **Meaning:** [What this color represents for your brand]
- **Accessibility:** Contrast ratio with base-100: [X.X:1]

**Color Variations:**
- Lighter: `oklch(75% 0.20 250)` - Hover states, backgrounds
- Darker: `oklch(45% 0.24 250)` - Active states, emphasis

---

#### Secondary Brand Color
```css
--color-secondary: oklch([L]% [C] [H]);
--color-secondary-content: oklch([L]% [C] [H]);
```

**Example:** `oklch(70% 0.18 50)` - Warm orange
- **Usage:** Secondary CTAs, accents, supporting elements
- **Meaning:** [What this color represents]
- **Accessibility:** Contrast ratio with base-100: [X.X:1]

---

#### Accent Color
```css
--color-accent: oklch([L]% [C] [H]);
--color-accent-content: oklch([L]% [C] [H]);
```

**Example:** `oklch(65% 0.20 300)` - Purple accent
- **Usage:** Highlights, badges, special features, notifications
- **Meaning:** [What this color represents]
- **Accessibility:** Contrast ratio with base-100: [X.X:1]

#### Neutral Colors

### Color Usage Guidelines

**Do's:**
- ✅ Use primary color for main CTAs and important actions
- ✅ Use secondary color to complement primary, not compete
- ✅ Use accent sparingly for emphasis and highlights
- ✅ Use semantic colors consistently (green = success, red = error)
- ✅ Ensure all text meets WCAG 2.1 AA contrast requirements
- ✅ Test colors in both light and dark modes (if applicable)

**Don'ts:**
- ❌ Don't use more than 3-4 colors in a single view
- ❌ Don't use color as the only indicator (add icons/text)
- ❌ Don't override semantic color meanings
- ❌ Don't use low-contrast color combinations
- ❌ Don't use pure black (#000000) for text (use base-content)

### Atom Components

**Component Template:**

```tsx
// components/atoms/[ComponentName]/[ComponentName].tsx
import React, { FC, HTMLAttributes } from 'react';

/**
 * Define the required and optional props for the component.
 * We extend HTMLAttributes<[element]> for easy access to standard DOM attributes 
 * like onClick, id, etc.
 */
export interface [ComponentName]Props extends HTMLAttributes<[element]> {
    // Required props
    /** The primary required property for this component. */
    [requiredProp]: [type]; 

    // Optional props
    /** An optional property with a specified default value. */
    [optionalProp]?: [type];

    // Common attributes (inherited from HTMLAttributes but can be specified here for clarity)
    /** Additional CSS classes to apply to the component. */
    className?: string;
    /** The content to be rendered inside the component. */
    children?: React.ReactNode;
}

/**
 * [ComponentName] is a versatile UI component built with React and TypeScript.
 * @param props - The properties object conforming to [ComponentName]Props.
 * @returns A React element.
 */
export const [ComponentName]: FC<[ComponentName]Props> = ({
    // Required prop - must be destructured
    [requiredProp],
    
    // Optional prop with a default value
    [optionalProp] = [defaultValue], 
    
    // Default values for common props
    className = '',
    children,
    
    // Collect all other standard HTML attributes
    ...props
}) => {
    // Determine the base classes for the component
    const baseClasses = '[base-classes]';
    
    return (
        <[element]
            className={`${baseClasses} ${className}`}
            {...props} // Spreads all remaining standard HTML attributes
        >
            {children}
        </[element]>
    );
};
```

**Example: Button Component**

```tsx
// components/atoms/Button/Button.tsx
import React from 'react';

export interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary' | 'outline' | 'ghost' | 'link';
  size?: 'sm' | 'md' | 'lg';
  loading?: boolean;
  disabled?: boolean;
  fullWidth?: boolean;
  icon?: React.ReactNode;
  iconPosition?: 'left' | 'right';
  children: React.ReactNode;
}

export const Button = ({
  variant = 'primary',
  size = 'md',
  loading = false,
  disabled = false,
  fullWidth = false,
  icon,
  iconPosition = 'left',
  children,
  className = '',
  type = 'button',
  ...props
}: ButtonProps) => {
  const variantClasses = {
    primary: 'btn-primary',
    secondary: 'btn-secondary',
    outline: 'btn-outline',
    ghost: 'btn-ghost',
    link: 'btn-link',
  };
  
  const sizeClasses = {
    sm: 'btn-sm',
    md: '',
    lg: 'btn-lg',
  };
  
  return (
    <button
      type={type}
      className={`btn ${variantClasses[variant]} ${sizeClasses[size]} ${fullWidth ? 'btn-block' : ''} ${className}`}
      disabled={disabled || loading}
      aria-busy={loading}
      aria-disabled={disabled || loading}
      {...props}
    >
      {loading && <span className="loading loading-spinner loading-sm" />}
      {!loading && icon && iconPosition === 'left' && <span className="mr-2">{icon}</span>}

};
```

### Molecule Components

**Example: FormField Component**

```tsx
// components/molecules/FormField/FormField.tsx
import React, { FC } from 'react';

export interface FormFieldProps extends Omit<InputProps, 'id'> {
  label: string;
  name: string;
  error?: string;
  helperText?: string;
  required?: boolean;
}

export const FormField: FC<FormFieldProps> = ({
  label,
  name,
  error,
  helperText,
  required = false,
  ...inputProps
}) => {
  const inputId = `field-${name}`;
  const errorId = error ? `${inputId}-error` : undefined;
  const helperId = helperText ? `${inputId}-helper` : undefined;
  
  return (
    <div className="form-control w-full">
      <label htmlFor={inputId} className="label">
        <span className="label-text">
          {label}
          {required && <span className="text-error ml-1" aria-label="required">*</span>}
        </span>
      </label>
      
      <Input
        id={inputId}
        name={name}
        required={required}
        error={!!error}
        aria-invalid={!!error}
        aria-describedby={[errorId, helperId].filter(Boolean).join(' ') || undefined}
        {...inputProps}
      />
      
      {(error || helperText) && (
        <label className="label">
          {error && (
            <span id={errorId} className="label-text-alt text-error" role="alert">
              {error}
            </span>
          )}
          {!error && helperText && (
            <span id={helperId} className="label-text-alt">
              {helperText}
            </span>
          )}
        </label>
      )}
    </div>
  );
};
```

**Creation Checklist:**

- [ ] Add responsive design classes
- [ ] Create test file with unit tests
- [ ] Document props and usage

**Naming Conventions:**

- **Files:** PascalCase (`Button.tsx`)
- **Components:** PascalCase (`Button`)

- **Variants:** lowercase with hyphens (`'primary'`, `'outline'`)

**Accessibility Requirements:**

- All interactive elements keyboard accessible
- Use semantic HTML elements
- Provide ARIA labels for icon-only buttons
- Ensure color contrast meets WCAG 2.1 AA
- Include focus visible states
- Support screen readers
